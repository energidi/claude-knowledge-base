## Restoring a Flow from a Metadata API backup after deploying unverified content over the live active version

**Context:** Same rollout as the entries in `flow-metadata.md`, ISP-6429 ICD-10 Lookup LWC (`projects/icd-lookup`) and its one consuming Screen Flow, `Community_Rare_eTRF_Page_2_Screen_Flow`. July 2026.

**Problem:** While troubleshooting the Flow-version deletions above, a local `.flow-meta.xml` file that was believed to be a verified backup of the Flow's real, in-use content (with all five ICD Lookup component instances correctly configured) was deployed back to the org as the new Active version - without re-checking its actual content immediately beforehand. It turned out that file no longer matched what had been verified earlier in the same session (the exact mechanism for how the local file's content changed was not conclusively identified) - it deployed successfully with no error, but silently went live missing the ICD Lookup components entirely, replacing them with older plain-text input fields. The deploy succeeding was not evidence the *content* was correct - only that the XML was well-formed and schema-valid.

**Solution:** A separate, independent copy of the verified-correct Flow version's content was still available from earlier in the session: a raw Tooling API `SELECT Metadata FROM Flow WHERE Id='<versionId>'` JSON response, saved to disk and already grep-verified (before any of the corrupted-file activity) to contain all five ICD Lookup instances with their correct `uniquenessKey` values. Since the Tooling API's `Metadata` field uses the same field names/structure as Metadata API Flow XML (just JSON-serialized, with object keys in place of alphabetically-ordered XML elements), a small script converted it directly into a valid `.flow-meta.xml` file: recursively walk the JSON, sort each object's keys alphabetically (matching the ordering the Salesforce CLI itself produces on retrieve), render arrays as repeated same-named elements, drop `null` values entirely, and wrap the result in `<Flow xmlns="http://soap.sforce.com/2006/04/metadata">...</Flow>`. The one manual correction needed afterward: the JSON's `apiVersion` was a bare number (`59`) and had to be reformatted to the `X.0` string form (`59.0`) the Metadata API expects. The converted file was grep-verified for the expected component references and validated as well-formed XML (via PowerShell's `[xml]` cast) *before* deploying, and after deploying, the live org's new Active version was independently re-queried via the same Tooling API `Metadata` field to confirm the actual deployed content - not just the deploy status - matched expectations.

**Why this works (and the lesson it cost a mistake to learn):**
- **A successful deploy status is not content verification.** Metadata API deploys validate schema/XML well-formedness and cross-references, not business-logic correctness or "is this the content I intended." The only way to know a Flow deploy actually shipped the right thing is to query the live result afterward and check for the specific expected content, the same way the input was checked beforehand.
- Keep raw API query results (e.g. `sf data query ... --json` output saved to a scratch file) around as an independent recovery source even after they've served their original diagnostic purpose - in this case, an incidental Tooling API dump made earlier for an unrelated verification step turned out to be the only remaining reliable copy of the correct Flow content once the "official" local backup file was found to be wrong.
- Before overwriting or redeploying any org metadata that has already caused one surprise in the same session, re-verify the *specific content* being deployed (e.g. `grep` for an expected unique string) immediately before the deploy action, not just once earlier in the session - state can change between when something was checked and when it is actually acted on.
