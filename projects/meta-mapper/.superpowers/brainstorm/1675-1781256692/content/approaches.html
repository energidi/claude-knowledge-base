Architecture Approach

One key decision isn't specified in CLAUDE.md: where does shared logic live? Multiple components need the same utilities (filter serialization, elapsed time, node filtering, namespace detection for export).

A

Shared LWC Service Modules (Recommended)

Create 2-3 @salesforce/lwc service modules (no HTML, JS-only) alongside the main components:

  • metaMapperFilters.js — sessionStorage schema, serialize/deserialize, validate
  • metaMapperUtils.js — elapsed time, filename sanitization, namespace detection
  • metaMapperNodeUtils.js — filter application logic shared by Tree + Graph

Why: Tree and Graph share filter state. If the logic lives in each, a bug fix needs two edits. Service modules also make unit testing the filter logic trivial - no component mount needed.

B

All Logic in metaMapperResults

metaMapperResults owns all shared state and exposes filtered node arrays to children via props. Tree and Graph receive pre-filtered data - they don't filter themselves.

Simpler - fewer files, no module boundary to maintain. Risk: metaMapperResults becomes very large (filter logic + serialization + tab state + PE handling + node selection). CLAUDE.md already gives it a lot of responsibilities.

C

Inline in Each Component

Each component handles its own filtering, serialization, and utilities inline. Simpler to read in isolation.

Problem: Filter validation logic (discard stale types) must live in one place to stay consistent. Duplicating sessionStorage read/write across Tree and Graph leads to silent divergence. Not recommended.
